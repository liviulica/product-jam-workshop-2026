# Architecture Overview

Handy is a free, open-source, fully-local push-to-talk dictation app. You press a shortcut, speak, and your words are transcribed on-device and pasted into whatever app currently has focus. No audio ever leaves your machine. It runs on macOS, Windows, and Linux, and is distributed under the MIT license (`Handy/README.md`, `Handy/AGENTS.md`).

The project's stated goal is to be "the most forkable" speech-to-text app rather than the most feature-complete one (`Handy/README.md`), so the codebase favors a small set of clearly-separated modules over deep abstraction.

## What Handy Is

- **Free and open source.** MIT-licensed, community-driven (`Handy/README.md`).
- **Fully local / private.** Transcription and silence detection run on your own hardware; nothing is sent to the cloud by default. The one exception is optional LLM post-processing, which you explicitly configure to call an external provider (`Handy/src-tauri/src/actions.rs`, `post_process_transcription`).
- **Cross-platform.** macOS (Intel + Apple Silicon), x64 Windows, x64 Linux (`Handy/README.md`).
- **Two transcription engine families:**
  - **Whisper** models (Small / Medium / Turbo / Large) via `transcribe-rs`'s `whisper-cpp` backend, with GPU acceleration where available.
  - **Parakeet** (and other ONNX models) via `transcribe-rs`, CPU-optimized with automatic language detection.

Both engines are pulled in through the `transcribe-rs` crate, which is compiled with backend features per platform: `whisper-metal` on macOS, `whisper-vulkan` + `ort-directml` on Windows, `whisper-vulkan` on Linux (`Handy/src-tauri/Cargo.toml`).

## High-Level Architecture

Handy is a Tauri v2 application. It pairs a React/TypeScript frontend with a Rust backend in a single desktop binary (`Handy/README.md`, `Handy/AGENTS.md`).

```
+-----------------------------------------------------------+
|  Frontend (React + TypeScript, Vite, Tailwind)            |
|  - Settings UI, onboarding, model selector, overlay       |
|  - src/App.tsx, src/main.tsx                              |
|  - Zustand stores + useSettings hook                      |
+------------------------------+----------------------------+
                               |
        commands (FE -> BE)    |   events (BE -> FE)
        via src/bindings.ts    |   via @tauri-apps/api event
                               |
+------------------------------v----------------------------+
|  Backend (Rust, Tauri core)                               |
|  - src-tauri/src/lib.rs  (app setup, plugins, managers)   |
|  - Managers: Audio / Model / Transcription / History      |
|  - TranscriptionCoordinator (serializes the pipeline)     |
|  - audio_toolkit (cpal recorder, rubato resampler, VAD)   |
+-----------------------------------------------------------+
```

### Frontend to backend: Tauri commands

The frontend never touches the OS, the microphone, or the ML engines directly. It calls Rust functions through Tauri's `invoke` mechanism. Those functions are plain Rust functions annotated with `#[tauri::command]` and `#[specta::specta]`, registered in `collect_commands![...]` in `Handy/src-tauri/src/lib.rs`.

`tauri-specta` generates a typed TypeScript bridge at `Handy/src/bindings.ts` (note the header: "This file was generated by tauri-specta. Do not edit this file manually."). The frontend imports `commands` from `@/bindings` and calls e.g. `commands.initializeShortcuts()`, `commands.hasAnyModelsAvailable()` (`Handy/src/App.tsx`). Each generated method wraps `TAURI_INVOKE("...")` and returns a `Result<T, string>` (`Handy/src/bindings.ts`).

The full command surface is the `collect_commands![...]` list in `Handy/src-tauri/src/lib.rs`: settings mutations (`shortcut::change_*`), model management (`commands::models::*`), audio device control (`commands::audio::*`), transcription/model lifecycle (`commands::transcription::*`), and history (`commands::history::*`).

### Backend to frontend: Tauri events

The backend pushes state changes back to the UI by emitting events. The frontend subscribes with `listen<T>(...)` from `@tauri-apps/api/event`. Examples in `Handy/src/App.tsx`:

- `recording-error` (e.g. `microphone_permission_denied`, `no_input_device`), emitted from `Handy/src-tauri/src/actions.rs`.
- `paste-error`, emitted when pasting the transcription fails.
- `model-state-changed` (e.g. `loading_failed`), emitted by the transcription manager.

Typed events also flow through `tauri-specta`: `collect_events![managers::history::HistoryUpdatePayload]` in `Handy/src-tauri/src/lib.rs` produces the `events` object in `Handy/src/bindings.ts` for history updates.

### Backend structure: the Manager pattern

On startup, `initialize_core_logic()` in `Handy/src-tauri/src/lib.rs` constructs the core managers, wraps each in an `Arc`, and registers them with Tauri's managed state via `app_handle.manage(...)`:

- `AudioRecordingManager` (`managers/audio.rs`) — microphone capture and device management.
- `ModelManager` (`managers/model.rs`) — model download and on-disk management.
- `TranscriptionManager` (`managers/transcription.rs`) — loads the selected engine and runs inference.
- `HistoryManager` (`managers/history.rs`) — persists transcripts and WAV recordings.

Command handlers and the pipeline pull these back out with `app.state::<Arc<...>>()`. Settings are persisted via `tauri-plugin-store` and read through `settings::get_settings(app)`.

`lib.rs` also wires up the rest of the app: the system tray (`TrayIconBuilder`), the recording overlay window, the `TranscriptionCoordinator`, the autostart manager, and a long list of Tauri plugins (single-instance, global-shortcut, clipboard-manager, updater, macos-permissions, etc.).

### Deferred initialization

Two pieces are intentionally **not** initialized at startup, to avoid triggering OS permission dialogs before the user is ready:

- **Enigo** (keyboard/mouse simulation for pasting) and **global shortcuts** are started only after onboarding/permissions complete. The frontend calls `commands.initializeEnigo()` and `commands.initializeShortcuts()` once `onboardingStep === "done"` (`Handy/src/App.tsx`; see the comments in `initialize_core_logic` in `Handy/src-tauri/src/lib.rs`).

## End-to-End Control Flow

The core loop is: **global shortcut -> record audio (with VAD) -> transcribe -> optional post-process -> paste into the focused app.** Here is where each step lives.

```
 [ Global shortcut / CLI flag / Unix signal ]
                  |
                  v
   shortcut/handler.rs  ->  coordinator.send_input(...)
                  |
                  v
  +---------------------------------------------+
  | TranscriptionCoordinator (single thread)    |   transcription_coordinator.rs
  | Idle -> Recording -> Processing -> Idle      |
  +----------------+------------------+----------+
        start()    |                  |  stop()
                   v                  v
        TranscribeAction::start   TranscribeAction::stop      actions.rs
                   |                  |
                   v                  v
   AudioRecordingManager        AudioRecordingManager.stop_recording()
   .try_start_recording()       -> Vec<f32> (16 kHz mono)
   (cpal capture + rubato                 |
    resample + Silero VAD)                v
                              TranscriptionManager.transcribe(samples)
                                          |        (Whisper / Parakeet)
                                          v
                              process_transcription_output()
                              (Chinese variant convert, optional LLM)
                                          |
                                          v
                              utils::paste(final_text)  ->  focused app
```

### 1. Trigger

A global keyboard shortcut is captured by the shortcut module. The handler (`Handy/src-tauri/src/shortcut/handler.rs`) routes transcribe bindings to the coordinator:

```rust
coordinator.send_input(binding_id, hotkey_string, is_pressed, settings.push_to_talk);
```

The same entry point is reachable without the keyboard:

- **CLI flags** on an already-running instance (`--toggle-transcription`, `--toggle-post-process`, `--cancel`) are delivered via `tauri_plugin_single_instance` and dispatched in `Handy/src-tauri/src/lib.rs`, which calls `signal_handle::send_transcription_input(...)`.
- **Unix signals** (`SIGUSR1`/`SIGUSR2`) are handled in `Handy/src-tauri/src/signal_handle.rs`, which also calls `send_transcription_input(...)`.

All of these funnel into `TranscriptionCoordinator::send_input` (`Handy/src-tauri/src/signal_handle.rs`, `Handy/src-tauri/src/shortcut/handler.rs`).

### 2. Coordinate (serialize the lifecycle)

`TranscriptionCoordinator` (`Handy/src-tauri/src/transcription_coordinator.rs`) owns a single background thread and a `mpsc` channel. Every input, cancel, and "processing finished" notification is processed sequentially by that one thread, which holds the pipeline `Stage` (`Idle` / `Recording` / `Processing`). This deliberately eliminates races between keyboard shortcuts, CLI flags, signals, and the async transcribe/paste task.

The thread debounces rapid presses (`DEBOUNCE = 30ms`), and supports both **push-to-talk** (press to start, release to stop) and **toggle** modes. When it decides to start or stop, it looks up the action in `ACTION_MAP` and calls `start()` / `stop()` (`Handy/src-tauri/src/transcription_coordinator.rs`, functions `start` and `stop`).

### 3. Record + VAD

`TranscribeAction::start` in `Handy/src-tauri/src/actions.rs`:

- Kicks off model load in the background (`tm.initiate_model_load()`) and pre-loads the VAD model (`rm.preload_vad()`) in parallel.
- Switches the tray icon and shows the recording overlay.
- Starts microphone capture via `AudioRecordingManager::try_start_recording(...)`.
- Plays the start feedback sound and optionally mutes other audio while recording.

The actual capture pipeline lives in `audio_toolkit`. The recorder uses **`cpal`** for cross-platform audio input, a **`rubato`**-based `FrameResampler` to convert the device's native sample rate down to Whisper's 16 kHz (`Handy/src-tauri/src/audio_toolkit/audio/recorder.rs`, `Handy/src-tauri/src/audio_toolkit/audio/resampler.rs`), and **Silero VAD** for silence detection.

Voice Activity Detection is attached to the recorder rather than run as a separate pass: `AudioRecordingManager` builds a `SileroVad` (threshold `0.3`) wrapped in a `SmoothedVad` and installs it on the recorder with `.with_vad(...)` (`Handy/src-tauri/src/managers/audio.rs`). The Silero model file (`resources/models/silero_vad_v4.onnx`) filters non-speech frames during capture, so the buffer that reaches transcription is already trimmed of silence. The VAD/Silero code is in `Handy/src-tauri/src/audio_toolkit/vad/` (`silero.rs`, `smoothed.rs`).

### 4. Stop + transcribe

`TranscribeAction::stop` in `Handy/src-tauri/src/actions.rs`:

- Unmutes audio, plays the stop sound, switches the tray to "transcribing".
- Spawns an async task (`tauri::async_runtime::spawn`) guarded by a `FinishGuard` that always notifies the coordinator (`notify_processing_finished`) when the task ends, even on panic.
- Calls `rm.stop_recording(binding_id)` (`Handy/src-tauri/src/managers/audio.rs`), which honors an optional trailing buffer, returns the captured `Vec<f32>` of 16 kHz samples, and pads very short clips.
- Saves the recording to a WAV file concurrently with transcription.
- Calls `tm.transcribe(samples)` (`Handy/src-tauri/src/managers/transcription.rs`, `transcribe`), which dispatches to the loaded engine — `WhisperEngine` or `ParakeetModel`/other ONNX models — applying language settings, optional translate-to-English, and custom-word prompting. Engine calls are wrapped in `catch_unwind` so an engine panic unloads the model instead of poisoning the shared mutex.

### 5. Post-process (optional LLM) + normalize

The raw transcription is passed to `process_transcription_output(...)` in `Handy/src-tauri/src/actions.rs`:

- **Chinese variant conversion**: if the selected language is `zh-Hans` / `zh-Hant`, it converts between Simplified and Traditional using OpenCC (`maybe_convert_chinese_variant`).
- **LLM post-processing** (only when the binding is `transcribe_with_post_process` / post-process is enabled): `post_process_transcription(...)` sends the text to the configured provider via `llm_client`, using structured-output JSON where supported, with a legacy `${output}` prompt-template fallback. On Apple Silicon macOS it can route to **Apple Intelligence** natively instead of an HTTP provider. This is the only step that may reach the network, and only when the user has configured it.

The result is a `ProcessedTranscription { final_text, post_processed_text, post_process_prompt }`, which is also saved to history via `HistoryManager::save_entry`.

### 6. Paste into the focused app

If `final_text` is non-empty, the async task hops to the main thread (`ah.run_on_main_thread(...)`) and calls `utils::paste(final_text, app)` (defined as `paste` in `Handy/src-tauri/src/clipboard.rs`). This inserts the text into whatever application currently has focus, using the configured paste method (clipboard paste, direct typing via Enigo, platform typing tools like `xdotool`/`wtype`/`dotool` on Linux, or an external script). On failure it emits `paste-error`, which the frontend turns into a toast. Finally the overlay is hidden and the tray returns to idle.

### Cancellation

A `cancel` binding, the `--cancel` CLI flag, and the tray "cancel" item all route to `utils::cancel_current_operation(app)`. That notifies the coordinator (`notify_cancel`), which resets the stage to `Idle` only when not mid-`Processing` (`Handy/src-tauri/src/transcription_coordinator.rs`). A dedicated cancel shortcut is registered while recording and unregistered when it stops (`Handy/src-tauri/src/actions.rs`).

## Frontend Lifecycle

`Handy/src/main.tsx` is the entry point: it sets a per-platform data attribute, initializes i18n, kicks off the model store, and renders `<App/>`. `Handy/src/App.tsx` drives onboarding (`accessibility` -> `model` -> `done`), wires up event listeners for backend errors, and — once onboarding is done — calls `initializeEnigo()` and `initializeShortcuts()` and refreshes audio devices. State is held in Zustand stores with a `useSettings` hook on top, and settings mutations go back to Rust through the generated `commands` bridge.

## Key Dependencies

Confirmed in `Handy/src-tauri/Cargo.toml`:

- **`transcribe-rs`** (features `whisper-cpp`, `onnx`; per-platform `whisper-metal` / `whisper-vulkan` / `ort-directml`) — Whisper and Parakeet/ONNX inference. (The README also refers to local Whisper inference as `whisper-rs`; in this codebase Whisper is reached through `transcribe-rs`'s `whisper_cpp` module.)
- **`cpal`** — cross-platform audio capture.
- **`vad-rs`** (Silero) — voice activity detection.
- **`rdev`** — low-level global keyboard/mouse events (used alongside `tauri-plugin-global-shortcut`).
- **`rubato`** — audio resampling to 16 kHz.
- **`enigo`** — keyboard simulation for pasting.
- **`rodio`** — audio feedback sounds.
- **`tauri-specta` / `specta`** — generate the typed `src/bindings.ts` command/event bridge.
